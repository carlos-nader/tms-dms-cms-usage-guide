DMS Down Research — Deep Dive into Dash-34 Date: 16 January 2026, ~11:18 -03 Status: RESEARCH IN PROGRESS — Preparing for design discussion KEY DASH-34 FINDINGS Section 2.1.6.3 — Sensor of Interest (SOI) The Dash-34 explicitly states that DMS Down toggles SOI between the HUD and MFDs:

DMS Down can toggle FROM HUD TO MFD.

DMS Down can toggle BETWEEN MFDs (from one MFD to the other).

The wording is “toggles”, not “cycles”, but in practice, when HUD is a valid SOI, repeated toggles produce a HUD → MFD → other MFD → HUD pattern.

This is the authoritative baseline behaviour for DMS Down.

Section 2.1.1.2.3 — SOI Constraints by Master Mode This section defines which displays can be SOI in each master mode, which constrains what DMS Down can actually toggle between:

NAV Master Mode Valid SOI: HUD, FCR, TGP, HSD, WPN, HAD. Implication for DMS Down: HUD is in the SOI set, so the conceptual cycle is: HUD → right MFD SOI → left MFD SOI → HUD → ...

A-A Master Mode Valid SOI: FCR, HSD, TGP only. HUD cannot be SOI. Implication for DMS Down:

HUD is out of the SOI set.

DMS Down is restricted to toggling between the two MFDs: right MFD SOI ↔ left MFD SOI.

A-G PRE (Preplanned A-G) Valid SOI: HUD, FCR, TGP, WPN, HAD, HSD. DMS Down can toggle between HUD and MFDs. The effective cycle is the same pattern as NAV, but which sensor is on which MFD depends on pilot setup: HUD → whichever MFD currently holds an A-G sensor → the other MFD → HUD.

A-G VIS (Visual A-G) Valid SOI: HUD, FCR, TGP, WPN. This is a subset of the A-G set and is visual-centric:

HUD is the primary visual designation interface (CCIP, DTOS, AGM-65 VIS, IAM-VIS, HUD/HMCS MARK).

DMS Down is the way to move SOI off the HUD and onto one of the MFDs (TGP/WPN/FCR) and then back again.

DGFT (Dogfight) Valid SOI: FCR, HSD, TGP. HUD cannot be SOI. DMS Down toggles only between MFDs (MFD↔MFD), similar to A-A.

MSL OVRD (Missile Override) Valid SOI: FCR, HSD, TGP. HUD cannot be SOI. Again, DMS Down toggles only between MFDs.

EXCEPTION STATES FOR DMS DOWN Snowplow (SP) PRE State Snowplow PRE is a special case where SP is entered but the Snowplow position has not yet been stabilised with TMS Up. In this state:

The A-G radar and TGP show NOT SOI.

The SOI is effectively “nowhere” — there is no active SOI display.

As a result, DMS Up and DMS Down have no effect until the SP position is stabilised.

Once the SP position is stabilised (TMS Up), SOI returns to its previous display and DMS Down resumes normal behaviour.

This applies symmetrically to DMS Up and DMS Down: both depend on there being a valid SOI candidate; when SOI is nowhere, neither direction can assign or toggle SOI.

MARK/OFLY Submode MARK/OFLY is a special submode documented under SOI behaviour. In this state:

The SOI cannot be designated at all.

Any DMS input that would normally change SOI (Up or Down) is ineffective.

For DMS Down, this means:

Pressing DMS Down in MARK/OFLY does not transfer SOI between HUD and MFDs or between MFDs.

SOI remains frozen on whatever display held it upon entering MARK/OFLY, until the mode changes.

There is no evidence in Dash‑34 of additional DMS‑Down‑only exception states beyond SP PRE and MARK/OFLY.

BEHAVIOUR SUMMARY BY CONTEXT (FOR 4.2.2 DESIGN) NAV Master Mode DMS Down is a convenience tool for moving SOI off the HUD onto an MFD and back.

Typical use: switch SOI between HUD (flight/navigation cues) and an MFD sensor (HSD/FCR/TGP) while cruising or navigating.

Behaviour pattern:

From HUD SOI → DMS Down moves SOI to the “next” MFD (commonly right).

Subsequent DMS Down → toggles to the other MFD.

Next DMS Down → returns SOI to HUD.

A-A, DGFT, MSL OVRD HUD is not a valid SOI candidate.

DMS Down becomes a pure MFD–MFD toggle:

If right MFD is SOI, DMS Down moves SOI to left MFD.

If left MFD is SOI, DMS Down moves SOI to right MFD.

This is critical for prioritising between FCR, HSD, and TGP in air‑to‑air contexts.

A-G PRE HUD is a valid SOI and part of the cycle.

DMS Down is again a convenience function: pilot can keep SOI on FCR/TGP/HSD/WPN/HAD as primary A‑G sensor, or temporarily move SOI to HUD if needed.

The exact mapping (which sensor is on which MFD) is configuration‑dependent, but the cycle always involves HUD and the two MFDs.

A-G VIS (CCIP, DTOS, AGM‑65 VIS, IAM‑VIS) In visual modes, DMS Down has operational importance comparable to DMS Up, because:

HUD is the primary visual designation interface (pipper, TD box, IAM TD, HUD/HMCS MARK).

Visual designation and rejection rely on HUD/HMCS line‑of‑sight and TMS/CURSOR behaviour with HUD as SOI.

However, sensor work (TGP fine search, WPN video cross‑check, FCR A‑G ranging outside CCIP, etc.) is performed on MFDs.

Therefore, in A-G VIS:

DMS Up: re‑establishes HUD SOI when SOI has migrated to an MFD.

DMS Down: does the opposite — moves SOI off HUD and onto an MFD so the pilot can work a sensor page hands‑on, then toggle back toward HUD.

The underlying cycle is the same (HUD ↔ MFD ↔ other MFD ↔ HUD), but in A‑G VIS the operational coupling between HUD and sensor pages makes DMS Down just as critical to flows as DMS Up.

PROPOSED HOTAS TABLE STRUCTURE (4.2.2.2) To keep the table parallel to DMS Up while capturing the distinct role of DMS Down, the core lines should be:

NAV

State: NAV.

Dir: Down.

Act: Short.

Function: Toggle SOI between HUD and MFDs.

Effect / Nuance: Describe the HUD → right MFD → left MFD → HUD cycle, with emphasis on navigation and SA management.

A-A

State: A‑A.

Dir: Down.

Act: Short.

Function: Toggle SOI between MFDs only.

Effect / Nuance: Explicitly state that HUD is excluded from being SOI and that DMS Down is the primary way of choosing which MFD (FCR/HSD/TGP) receives HOTAS commands.

A-G

State: A‑G.

Dir: Down.

Act: Short.

Function: Toggle SOI between HUD and A‑G sensor pages.

Effect / Nuance:

In preplanned A‑G, this is mostly convenience.

In A‑G VIS modes, it is operationally critical to alternating control between HUD visual cueing and MFD sensor management.

TRAINING MISSION MAPPING (RATIONALE) A-G (especially VIS) TRN 10 – GP Bombs:

Uses CCIP and DTOS, including HUD‑centric visual deliveries.

The mission narrative has the pilot repeatedly set A‑G master mode, SMS, CCRP/CCIP/DTOS, and manage HUD SOI for delivery.

DMS Down is naturally employed to move SOI from HUD to an MFD (e.g., HSD or FCR/A‑G) when changing profiles or checking ranging.

TRN 11 – LGB:

Combines HUD symbology and TGP on an MFD, with targeting via TGP and releases guided by HUD cues.

DMS Down is used to shift SOI between TGP and HUD as needed.

TRN 13 / 14 – Maverick Basic/Advanced:

For PRE/VIS modes, there is a constant alternation between HUD/TGP/WPN.

The Training Manual explicitly mentions using DMS Down to bring HSD SOI before using TMS to select steerpoints (HARM example by analogy, but the pattern is identical for other weapons).

VIS and HUD/HMCS MARK flows strongly exercise SOI toggling.

TRN 15 – IAM:

JDAM/JSOW profiles often require moving SOI between HUD and HSD/TGP to manage SPI, impact azimuth, etc.

DMS Down is part of the “sensor management” pattern even though the mission focuses more on IAM specifics.

Given this, reusing TRN 10, 11, 13, 14, 15 in the Train column for the A‑G DMS Down row is well‑justified: these missions are the natural playground for learning HUD↔MFD SOI toggling under A‑G VIS and preplanned conditions.

A-A Context For the A‑A DMS Down row, adding A‑A missions is justified because:

TRN 18 – BARCAP (Radar Missiles):

The Training Manual describes extensive FCR/HSD use, sort and targeting.

Managing which display is SOI (FCR vs HSD) is essential to BVR flow; DMS Down is the primary HOTAS method for toggling SOI between these MFDs in A‑A.

TRN 17B – IFF Intercept:

Emphasises radar usage and IFF interrogation; again, FCR SOI selection is key.

The mission expects the pilot to interrogate and sort using FCR/HSD; SOI is frequently moved between displays.

Thus, a reasonable mapping is:

A‑A row Train column: rnref{18 (BARCAP)}, rnref{17B (IFF Intercept)}.

A‑G row Train column: rnref{10 (GP Bombs)}, rnref{11 (LGB)}, rnref{13 (Maverick)}, rnref{14 (Maverick Adv)}, rnref{15 (IAM)}.

NUANCES AND LIMITATIONS There is no evidence in Dash‑34 of additional DMS‑Down‑specific exception states beyond Snowplow PRE and MARK/OFLY.

SWAP OSB behaviour is related but orthogonal: when a format is swapped between MFDs, the SOI border follows the format; this is already covered at conceptual level in 4.1 and MFD sections and does not need duplication in 4.2.2.

DMS Left/Right is clearly independent from DMS Down: Left/Right changes which formats are loaded on a given MFD, but does not change SOI.
